# 点之出众企业内部 Agent Roadmap

## 背景

点之出众当前希望将本项目从通用本地 Agent 工作台，逐步改造成企业内部运营执行平台。目标使用规模为 1 名运维管理人员和约 10 名运营人员，主要入口为飞书，本地管理后台用于开发、配置、排障和审计。

当前系统已经具备飞书连接器、Tab 会话、工作提示词、Skill、记忆、定时任务、浏览器工具和消息队列等基础能力。下一阶段的重点不是继续堆叠工具，而是建立企业内部使用需要的身份、权限、历史、记忆、任务、并发和审计边界。

## 目标用户和职责

### 运维管理人员

运维管理人员负责系统可用性、配置、权限和风险控制。

- 管理飞书应用、模型服务、浏览器 Profile、商家后台登录态。
- 安装和维护 Skill、browser-act 能力、商家后台操作 Recipe。
- 管理运营人员、角色、门店范围和高风险操作权限。
- 查看所有任务、失败原因、执行日志、截图证据和模型用量。
- 配置审批规则、并发策略、资源锁和异常告警。
- 维护企业公共记忆、SOP、门店操作规范和团队知识库。

### 运营人员

运营人员通过飞书发起业务任务，主要关注门店运营、后台操作、截图、诊断和报表。

- 在飞书私聊或群聊中发起门店相关任务。
- 查看自己发起的任务状态、执行结果和截图证据。
- 在授权范围内操作自己负责的门店。
- 对高风险动作进行明确确认。
- 使用群聊协作处理同一批门店、同一类问题或同一阶段活动。

## 当前能力基线

### 已有能力

- 飞书消息可映射为 connector Tab。
- 每个 Tab 有独立 AgentRuntime。
- 同一 Tab 内消息可排队串行处理。
- 不同 Tab 可并行处理。
- Tab 可拥有独立记忆文件。
- 飞书默认工作提示词可通过 `feishu_default_work_prompt` 注入。
- 支持 `/status`、`/stop`、`/memory`、`/merge-memory`、`/clone` 等基础命令。
- 支持定时任务、Skill 扩展、文件操作、浏览器操作和飞书文档工具。

### 当前限制

- 用户身份只停留在飞书 pairing 和 connector source 层，没有完整员工档案。
- 记忆主要按 Tab 隔离，没有企业公共记忆、员工个人记忆、群聊记忆三层结构。
- 历史主要是会话消息，不是任务化历史，难以复盘和审计。
- 群聊多人并发时使用同一个 Tab 队列，任务之间缺少独立状态。
- 不同私聊 Tab 可并行执行，但缺少门店级、浏览器 Profile 级和商家后台账号级资源锁。
- 商家后台操作缺少风险分级、审批流和证据归档。
- 浏览器 Profile 和登录态还没有形成运维统一管理界面。

## 目标架构

### 核心对象

企业内部使用需要围绕以下对象建模。

| 对象 | 说明 |
| --- | --- |
| Employee | 内部员工，来自飞书 userId/openId，包含姓名、角色、状态 |
| Role | 角色，例如运维管理员、运营人员、只读观察者 |
| StoreScope | 门店授权范围，限制员工可查看或操作的门店 |
| Conversation | 飞书私聊或群聊会话，绑定 connectorId、conversationId、chatType |
| MemoryScope | 记忆作用域，包含企业公共、员工个人、群聊会话、任务临时记忆 |
| Task | 一次业务请求或后台操作的任务记录 |
| TaskEvent | 任务执行过程中的事件、步骤和错误 |
| Artifact | 截图、文件、飞书文档、浏览器轨迹等证据 |
| ResourceLock | 门店、浏览器 Profile、商家后台账号等资源锁 |
| Approval | 高风险操作的确认和审批记录 |

### 记忆分层

建议将记忆拆成四层，并在构造飞书端上下文时按需注入。

| 记忆层 | 作用 | 示例 |
| --- | --- | --- |
| 企业公共记忆 | 所有人共享的长期规则和 SOP | 美团后台操作规范、图片上传标准、活动配置规则 |
| 员工个人记忆 | 某个运营人员的职责、偏好和负责门店 | 张三负责 A/B/C 门店，常处理美团装修 |
| 群聊记忆 | 某个飞书群的协作上下文 | 本群正在处理 6 月美团门店整改 |
| 任务临时记忆 | 单次任务执行中的中间状态 | 已登录、已截图、等待用户确认发布 |

飞书消息进入 Agent 时，建议按下面顺序组装上下文：

```text
企业公共记忆
+ 当前员工个人记忆
+ 当前私聊/群聊记忆
+ 当前任务历史摘要
+ 当前消息
```

这样可以避免把所有长期信息塞进一个 Tab 记忆，也能避免同一个运营人员在不同群聊里丢失个人上下文。

### 历史分层

历史不应只保存聊天记录，应拆成对话历史和任务历史。

对话历史保留原始消息：

- connectorId
- conversationId
- chatType
- senderId
- senderName
- messageId
- messageText
- receivedAt
- agentReply
- repliedAt

任务历史记录可执行事项：

- taskId
- 发起人
- 来源群聊或私聊
- 平台，例如美团、饿了么、抖音
- 门店 ID 和门店名称
- 任务类型，例如截图、诊断、修改电话、上传图片、改菜单
- 风险等级
- 状态
- 资源锁
- 确认人和确认时间
- 执行日志
- 截图和文件证据
- 失败原因和重试记录

## 并发和队列策略

### 当前策略

当前系统基本策略是：

```text
同一个 Tab 串行排队
不同 Tab 可并行执行
```

飞书 connector Tab 自带 `pendingMessages` 和 `processingMessageId`，单个 Tab 最多排队 5 条。Gateway MessageHandler 还会根据 `runtime.isCurrentlyGenerating()` 对同一个 session 再做串行队列。

### 企业场景目标策略

企业内部运营场景需要从消息队列升级为任务队列。

```text
飞书消息进入
-> 识别员工、群聊、门店、任务类型、风险等级
-> 创建 Task
-> 判断权限和资源锁
-> 可并发任务直接执行
-> 需要锁或审批的任务排队/等待确认
-> 写入任务事件和证据
-> 回复飞书
```

### 资源锁

建议优先实现以下资源锁。

| 锁 | 作用 |
| --- | --- |
| `employee:{id}` | 同一员工的高风险操作可串行 |
| `conversation:{id}` | 同一群聊内需要连续上下文的任务可串行 |
| `store:{platform}:{storeId}` | 同一门店的修改操作必须串行 |
| `browser-profile:{profileId}` | 同一个浏览器 Profile 避免被多个任务同时操作 |
| `merchant-account:{platform}:{accountId}` | 同一个商家后台账号避免并发冲突 |

只读任务可以不拿门店写锁，但后台修改、保存、发布必须拿锁。

### 风险分级

| 等级 | 示例 | 策略 |
| --- | --- | --- |
| L0 只读 | 查询、截图、读取页面、核查配置 | 可并发，直接执行 |
| L1 低风险 | 生成草稿、整理资料、不生效配置 | 可执行前简单确认 |
| L2 高风险 | 修改电话、营业时间、价格、上传并生效图片 | 必须明确确认，按门店和浏览器串行 |
| L3 敏感 | 删除、批量发布、多门店批量变更 | 运维或管理员审批 |

## 权限和风控

### 员工权限

建议建立员工表，至少包含：

- 飞书 openId/userId
- 姓名
- 角色
- 是否启用
- 是否运维管理员
- 负责门店范围
- 可执行任务类型
- 可审批风险等级

### 门店权限

运营人员只能对授权门店发起后台操作。未授权门店可允许查询公开信息，但禁止后台修改。

### 操作确认

模型提示词只能作为第一层约束，关键动作必须由代码层判断。

高风险操作在工具执行前应生成确认请求：

```text
将要把「朝阳示例门店」的联系电话从 010-88886666 修改为 13800000000。
该操作会保存到美团商家后台。
请回复“确认修改”继续。
```

确认记录写入 Approval 表，后续审计可追踪。

## 管理后台 Roadmap

### 员工管理

- 展示飞书用户、角色、启用状态。
- 设置运维管理员和运营人员。
- 配置门店范围和操作权限。
- 查看员工个人记忆和任务历史。

### 群聊管理

- 展示飞书群、群成员、群用途。
- 设置群聊默认工作提示词。
- 查看群聊记忆和群任务历史。
- 支持生成群日报或周报。

### 门店管理

- 管理平台、门店 ID、门店名称、负责人。
- 配置门店所在商家后台账号和浏览器 Profile。
- 查看门店历史任务、截图和异常记录。

### 浏览器环境管理

- 运维统一配置浏览器 Profile。
- 展示 Profile 登录状态、最近使用时间、可操作平台。
- 标记登录过期、需要扫码、可用、不可用。
- 避免运营人员在飞书端看到浏览器创建和 Profile 导入流程。

### 任务中心

- 展示所有任务状态：待执行、排队中、执行中、等待确认、已完成、失败。
- 支持按员工、群聊、门店、平台、风险等级筛选。
- 支持查看任务事件、模型回复、工具调用、截图证据。
- 支持取消、重试、转人工、升级审批。

### 审批中心

- 展示高风险待确认任务。
- 支持员工本人确认和运维管理员审批。
- 记录审批人、审批时间、审批内容和最终结果。

### 记忆中心

- 管理企业公共记忆。
- 管理员工个人记忆。
- 管理群聊记忆。
- 支持记忆编辑、合并、清理、版本记录。

### 运维看板

- 展示任务量、成功率、失败率、平均耗时。
- 展示模型用量、费用估算、异常工具调用。
- 展示浏览器登录态、队列积压和资源锁状态。
- 展示需要人工处理的失败任务。

## 分阶段计划

### Phase 0：稳定当前二开版本

目标：让当前系统以点之出众品牌和飞书入口稳定使用。

- 完成可见品牌替换和飞书默认工作提示词。
- 保留现有 Tab 记忆和消息队列机制。
- 运维手动配置飞书、模型、浏览器和 Skill。
- 运营通过飞书发起轻量任务。

验收标准：

- 飞书消息回复不暴露浏览器创建和底层工具流程。
- 管理后台显示点之出众品牌。
- 类型检查通过，服务可启动。

### Phase 1：员工、群聊、记忆分层

目标：解决不同员工和群聊的上下文隔离问题。

- 新增员工档案和角色。
- 将飞书 userId/openId 绑定到员工。
- 新增企业公共记忆、员工个人记忆、群聊记忆。
- 飞书消息进入时注入三层记忆。
- 管理后台增加基础员工和记忆查看功能。

验收标准：

- 同一员工在私聊和群聊中共享个人记忆。
- 不同员工个人记忆互不串线。
- 同一个群有独立群聊记忆。
- 企业公共记忆可被所有飞书会话使用。

### Phase 2：任务历史和证据归档

目标：从聊天记录升级到任务化历史。

- 新增 Task、TaskEvent、Artifact 数据表。
- 飞书消息可识别并创建任务。
- 截图、文件、日志、飞书文档链接归档为 Artifact。
- 管理后台增加任务中心初版。
- 支持按员工、群、门店查询任务。

验收标准：

- 每次后台操作或截图任务都有 taskId。
- 任务状态可追踪。
- 执行截图和关键日志可回看。
- 群聊中可查询历史任务摘要。

### Phase 3：权限、风控、审批

目标：减少商家后台误操作风险。

- 新增门店授权范围。
- 新增风险等级识别。
- L2/L3 操作必须确认或审批。
- 关键工具执行前检查权限和审批状态。
- 管理后台增加审批中心。

验收标准：

- 未授权运营不能修改非负责门店。
- 高风险动作没有确认不能执行。
- 审批记录可追踪。
- 飞书端能收到等待确认、已确认、已完成的状态提示。

### Phase 4：资源锁和持久化队列

目标：支撑 10 名运营同时使用，避免后台操作冲突。

- 新增任务队列表和资源锁表。
- 同一门店修改操作串行。
- 同一浏览器 Profile 操作串行。
- 只读任务允许并发。
- 服务重启后可恢复待执行任务和失败任务。

验收标准：

- 同一门店并发修改不会同时执行。
- 同一浏览器 Profile 不会被多个后台任务抢占。
- 队列积压可在后台查看。
- 重启后可恢复未完成任务状态。

### Phase 5：运维看板和自动化运营

目标：让系统具备持续运营能力。

- 增加模型用量、任务成功率、失败原因统计。
- 增加浏览器登录态监控。
- 增加群聊日报和周报。
- 增加门店异常定时扫描。
- 支持失败任务自动重试和人工接管。

验收标准：

- 运维可以快速发现登录过期、队列积压和失败任务。
- 群聊可自动收到每日运营摘要。
- 常见异常可自动生成处理建议。

## 技术实现建议

- 优先基于当前 SQLite 配置库扩展，不引入外部数据库。
- 先实现数据结构和管理后台只读视图，再接入自动调度。
- 风控判断必须在工具调用前做代码层校验，不能只依赖提示词。
- 资源锁需要设置过期时间，避免任务异常退出后永久占锁。
- 任务事件和截图证据应尽量轻量保存，避免快速膨胀。
- 记忆更新应支持版本记录和手动回滚。

## 风险

- 如果没有资源锁，多运营并发操作同一后台账号可能导致页面串台。
- 如果没有任务历史，后续很难追踪谁发起了什么变更。
- 如果只靠提示词做风控，模型仍可能在边界场景误执行。
- 如果所有群聊共享单一 Tab 记忆，长期使用后上下文会变得嘈杂。
- 如果浏览器登录态由运营临时配置，飞书端体验会不稳定。

## 推荐近期行动

近期优先做以下 5 件事：

1. 新增员工和角色模型，绑定飞书用户。
2. 新增企业公共记忆、员工个人记忆、群聊记忆。
3. 新增任务历史表，并让飞书端后台操作生成 taskId。
4. 新增浏览器 Profile 运维配置和状态展示。
5. 新增门店级和浏览器 Profile 级资源锁。

这 5 件完成后，系统就能从“飞书接入的本地 Agent”升级为“可管理、可追踪、可并发控制的内部运营执行 Agent”。
